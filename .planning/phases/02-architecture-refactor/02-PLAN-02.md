# Plan 2: Extract All Tool Handlers to Individual Files

## Objective

Move every tool from the `executeTool()` switch statement in `tools.ts` into its own handler file under `src/tools/handlers/`, each implementing the `ToolHandler` interface from Plan 1. After this plan, the switch statement in `tools.ts` is empty — all tool logic lives in handler files.

## Context

- Plan 1 created `ToolHandler`/`ToolContext` in `src/tools/types.ts`, media in `src/tools/media.ts`, security in `src/utils/security.ts`, and `stripHtml` in `src/utils/html.ts`.
- Each handler file exports a default `ToolHandler` or `ToolHandler[]` (for grouped tools like reminders, tasks, knowledge, workflows).
- The Zod schemas from `tools.ts` lines 17-40 move into their respective handler files. The `TOOL_SCHEMAS` map is eliminated.
- Each handler's `definition` property contains the Claude API tool definition (currently in the `allToolDefinitions` array).
- Each handler's `execute` method contains the case body from the switch statement.
- Dependencies like `config`, `db`, `addPendingMedia` come from `ToolContext` parameter — NOT from module-level imports. This is critical for testability.
- However, utility imports (`isPathAllowed`, `stripHtml`, `isShellCommandSafe`, `sanitizeWebContent`, `gitEnv`, `redactSecrets`) remain as direct imports from their utility modules.
- **Security preservation:** Every security check (shell blocklist, path validation, URL validation, email whitelist, git env, secret redaction) must be preserved exactly as-is in the handler files.
- Tools with `localOnly: true`: `screenshot`, `manage_project`, `send_file` (from `LOCAL_ONLY_TOOLS` array).
- **Important:** During this plan, `tools.ts` still has the `executeTool()` function and switch statement but it delegates to handler files via a temporary import pattern. The full registry wiring happens in Plan 3.

## Task 6: Gut the switch statement in tools.ts

After all handlers are created, the switch statement in `executeTool()` can be emptied. For now (before Plan 3 wires the registry), create a **temporary bridge** in `tools.ts`:

1. Import all handler files at the top of `tools.ts`
2. Build a `Map<string, ToolHandler>` from the imported handlers
3. Replace the switch statement body with a map lookup:

```typescript
// Temporary bridge — replaced by registry in Plan 3
import shellHandler from '../tools/handlers/shell.js';
import filesHandler from '../tools/handlers/files.js';
// ... all other handler imports ...

const handlerMap = new Map<string, ToolHandler>();
function registerHandlers(handlers: ToolHandler | ToolHandler[]) {
  const arr = Array.isArray(handlers) ? handlers : [handlers];
  for (const h of arr) handlerMap.set(h.name, h);
}
registerHandlers(shellHandler);
registerHandlers(filesHandler);
// ... all registrations ...

// In executeTool(), replace switch with:
const handler = handlerMap.get(name);
if (!handler) return `Unknown tool: ${name}`;
const ctx: ToolContext = { config, db, addPendingMedia };
return handler.execute(input, ctx);
```

Also replace the `allToolDefinitions` array with:
```typescript
export const toolDefinitions = Array.from(handlerMap.values()).map(h => h.definition);
```

The Zod validation at the top of `executeTool()` now uses `handler.schema` instead of `TOOL_SCHEMAS[name]`. Delete the `TOOL_SCHEMAS` map and all standalone Zod schema declarations from `tools.ts`.
