# Plan 1: Infrastructure — Types, Media, Security, HTML Extraction

## Objective

Create all foundational pieces that the tool handler extraction depends on: the `ToolHandler` interface, the media side-channel module, shared security and HTML utilities, and static HTML view files. After this plan, `tools.ts` shrinks by ~70 lines and `server.ts` shrinks by ~290 lines, with zero behavior change.

## Context

- The research (02-RESEARCH.md Section 1) defines the `ToolHandler` and `ToolContext` interfaces.
- Security functions (`isPathAllowed`, `isUrlAllowed`, `isEmailAllowed`) and their constants live in `tools.ts` lines 62-108. They must move to `src/utils/security.ts` so individual handler files can import them.
- Media side-channel (`pendingMediaMap`, `setCurrentRequestId`, `addPendingMedia`, `collectMedia`) lives in `tools.ts` lines 42-58. It must become its own module because 5+ tool handlers depend on `addPendingMedia` via `ToolContext`.
- HTML stripping is duplicated three times (research Section 4.1). It consolidates into `src/utils/html.ts`.
- Dashboard and chat HTML are self-contained static pages (research Section 3.1) — no server-side template variables. Extraction is a straight copy.
- Phase 1 added `src/utils/shell-blocklist.ts`, `src/utils/sanitize.ts`, `src/utils/git-auth.ts`. These are NOT moved — they stay in place and individual handlers will import from them directly.
- The `views/` directory contains `.html` files that tsc will NOT copy to `dist/`. A `postbuild` script is needed.

## Task 4: Create src/utils/html.ts with stripHtml

```typescript
// src/utils/html.ts

/**
 * Strip HTML tags, scripts, and styles from raw HTML.
 * Returns plain text with collapsed whitespace.
 */
export function stripHtml(html: string): string {
  return html
    .replace(/<script[\s\S]*?<\/script>/gi, '')
    .replace(/<style[\s\S]*?<\/style>/gi, '')
    .replace(/<[^>]+>/g, ' ')
    .replace(/\s+/g, ' ')
    .trim();
}
```

Update three call sites:

1. **`src/agent/tools.ts` — `browse_url` case** (~line 425-431): Replace the 4-line regex chain with `stripHtml(html)`. Keep the subsequent `.slice(0, 8000)` and `sanitizeWebContent()` calls.

3. **`src/agent/knowledge.ts` lines 98-103**: Replace with:

```typescript
import { readFileSync } from 'fs';
import { join } from 'path';

// Cache HTML at startup (no server-side variables needed)
const dashboardHTML = readFileSync(join(import.meta.dirname, '../views/dashboard.html'), 'utf-8');
const chatHTML = readFileSync(join(import.meta.dirname, '../views/chat.html'), 'utf-8');

app.get('/dashboard', dashAuth, (_req, res) => {
  res.setHeader('Content-Type', 'text/html; charset=utf-8');
  res.send(dashboardHTML);
});

app.get('/chat', dashAuth, (_req, res) => {
  res.setHeader('Content-Type', 'text/html; charset=utf-8');
  res.send(chatHTML);
});
```

Note: `import.meta.dirname` is available in Node 22 (the project's runtime). The path `../views/` is relative to `dist/gateway/server.js` and resolves to `dist/views/`.

## Task 6: Add postbuild copy step for views

HTML files are not TypeScript, so `tsc` won't copy them. Add to `package.json` scripts:

```json
"postbuild": "cp -r src/views dist/views"
```

The full scripts section becomes:
```json
"scripts": {
  "dev": "tsx watch src/index.ts",
  "build": "tsc",
  "postbuild": "cp -r src/views dist/views",
  "start": "node dist/index.js"
}
```

Also verify the Dockerfile (if it runs `npm run build`) will pick this up automatically — npm runs `postbuild` after `build` by convention.
