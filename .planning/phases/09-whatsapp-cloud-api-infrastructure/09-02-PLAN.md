# Plan 09-02: Wire Cloud API Webhook to Conversation Handler

## Objective

Route incoming Cloud API webhook messages through the existing message batcher and AI conversation handler. Right now the webhook POST only logs messages — they never reach the bot's brain. This plan adds a `CloudBotAdapter` that implements `BotAdapter` using `sendCloudMessage`, and wires it into the webhook handler so Cloud API messages flow through the full conversation pipeline.

## Context

- `src/whatsapp/cloud-api.ts` — `sendCloudMessage` + `parseWebhookPayload` (built in 09-01)
- `src/http/routes/whatsapp-cloud-webhook.ts` — webhook POST logs messages only
- `src/whatsapp/message-handler.ts` — `setupMessageHandler` for whatsapp-web.js client
- `src/whatsapp/message-batcher.ts` — `addMessageToBatch` (8-second debounce)
- `src/ai/conversation.ts` — `handleConversation` + `sendFirstMessage`
- `src/whatsapp/connection.ts` — `BotAdapter` type definition
- `src/follow-up/follow-up-db.ts` — `cancelFollowUps`

## Tasks

### Task 1: Create CloudBotAdapter in cloud-api.ts

Add a `CloudBotAdapter` class/factory to `src/whatsapp/cloud-api.ts` that implements the `BotAdapter` interface using `sendCloudMessage`. The adapter needs:
- `sendMessage(jid, { text })` — sends via Cloud API
- `sendPresenceUpdate(state, jid)` — no-op (Cloud API has no typing indicators)
- `sendAudio(jid, buffer, ptt)` — no-op (not supported in phase 9)
- `sendImage(jid, buffer, caption)` — no-op (not supported in phase 9)
- `sendDocument(jid, buffer, filename, caption)` — no-op (not supported in phase 9)

The adapter must accept a `phoneNumberId` at construction time so the correct WhatsApp number is used for replies.

**Done criteria:**
- `createCloudAdapter(phoneNumberId)` is exported from `cloud-api.ts`
- The adapter's `sendMessage` calls `sendCloudMessage` with correct params
- TypeScript compiles without errors

### Task 2: Wire webhook POST to conversation handler

Modify `src/http/routes/whatsapp-cloud-webhook.ts` to route incoming messages through the conversation pipeline:

1. For each parsed message: create `CloudBotAdapter` with the message's `phoneNumberId`
2. Cancel any pending follow-ups for the sender (like `message-handler.ts` does)
3. Call `addMessageToBatch(senderPhone, text, callback)` where callback calls `handleConversation`
4. Skip admin phone (isAdminPhone) — don't process boss messages via Cloud API webhook
5. Continue to always return 200 regardless

The webhook should NOT import from `connection.ts` (whatsapp-web.js) — it uses Cloud API only.

**Done criteria:**
- Incoming Cloud API messages route to `handleConversation`
- Follow-ups are cancelled on message receipt
- Admin phone messages are skipped
- Webhook always returns 200
- TypeScript compiles without errors

### Task 3: Tests for webhook → conversation routing

Add tests to `tests/whatsapp-cloud-routing.test.ts` (existing file) for the new wiring:

1. Test that `createCloudAdapter` calls `sendCloudMessage` with the correct `phoneNumberId`
2. Test that the webhook POST routes a text message to `addMessageToBatch`
3. Test that the webhook POST skips admin phone messages
4. Test that follow-ups are cancelled when a message arrives

**Done criteria:**
- New tests pass
- Existing tests remain green
- `npm test` runs without new failures (pre-existing failures are out of scope)
