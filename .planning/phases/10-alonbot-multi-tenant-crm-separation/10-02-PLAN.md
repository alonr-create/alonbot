# Plan 10-02: Wire Tenant Context Through Cloud Webhook Into Conversation Handler

## Objective

The `tenants` table and `lookupTenantByPhoneNumberId` already exist but the cloud webhook still calls `handleConversation` without tenant context. The conversation handler and system prompt builder fall back to the global singleton `tenant_config` table (Alon.dev only). This plan threads a `TenantRow` through the call chain so each Cloud API message is handled with the correct tenant's business name, admin phone, service catalog, and personality.

## Context

- `src/whatsapp/cloud-api.ts` — `createCloudAdapter`, `parseWebhookPayload`; adapter currently uses global `WA_CLOUD_TOKEN`
- `src/http/routes/whatsapp-cloud-webhook.ts` — calls `isAdminPhone(phone)` globally, not per tenant
- `src/ai/conversation.ts` — `handleConversation(phone, messages, adapter)` — no tenant param
- `src/ai/system-prompt.ts` — `buildSystemPrompt(name, interest, phone)` — uses global `getConfig`/`getOwnerName` etc.
- `src/db/tenants.ts` — `lookupTenantByPhoneNumberId(phoneNumberId, db?)` — returns `TenantRow | null`
- `src/db/tenant-config.ts` — `isAdminPhone(phone, tenant?)` — already accepts optional `TenantRow`
- `src/whatsapp/cloud-api.ts` — `sendCloudMessage` — reads token from env; needs per-tenant token support

## Tasks

### Task 1: Update handleConversation to accept optional TenantRow

Add an optional fourth parameter `tenant?: TenantRow` to `handleConversation`. When provided:
- Pass `tenant` to `isAdminPhone(phone, tenant)` calls
- Pass `tenant` to `buildSystemPrompt`
- Use `tenant.monday_board_id` for Monday.com updates instead of deriving from lead

**Done criteria:**
- `handleConversation(phone, messages, adapter, tenant?)` compiles
- `isAdminPhone` calls inside `handleConversation` pass the tenant
- TypeScript compiles without errors

### Task 2: Update buildSystemPrompt to accept optional TenantRow

Add optional `tenant?: TenantRow` to `buildSystemPrompt`. When provided, use tenant fields
(`business_name`, `owner_name`, `service_catalog`, `portfolio`, `sales_faq`, `sales_objections`,
`personality`, `timezone`) instead of global `getConfig` getters.

**Done criteria:**
- `buildSystemPrompt(name, interest, phone?, isWebsite?, tenant?)` compiles
- When `tenant` is supplied, prompt uses tenant fields (not global singletons)
- When `tenant` is null/undefined, behavior is unchanged (backward compatible)
- TypeScript compiles without errors

### Task 3: Wire tenant lookup into cloud webhook

Modify `src/http/routes/whatsapp-cloud-webhook.ts`:
1. For each parsed message: call `lookupTenantByPhoneNumberId(msg.phoneNumberId)` to resolve tenant
2. Use `isAdminPhone(msg.senderPhone, tenant ?? undefined)` — tenant-aware admin check
3. Pass the tenant to `handleConversation` via the batcher callback
4. If `tenant?.wa_cloud_token` is set, the `createCloudAdapter` should use it (pass token to adapter)

Update `createCloudAdapter` in `src/whatsapp/cloud-api.ts` to accept an optional `token` parameter
so per-tenant Cloud API tokens work (falls back to global env vars when not provided).

**Done criteria:**
- Webhook resolves tenant from `phoneNumberId` before processing each message
- `isAdminPhone` uses tenant when available
- `handleConversation` receives tenant
- `createCloudAdapter` accepts optional token
- TypeScript compiles without errors

2. Webhook with alondev `phoneNumberId` (967467269793135) routes to alondev tenant
