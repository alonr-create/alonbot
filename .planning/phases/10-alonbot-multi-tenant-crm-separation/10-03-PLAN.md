# Plan 10-03: Tenant-Aware Booking Routing

## Objective

Replace hardcoded "Dekel" detection in the conversation booking flow with tenant-aware routing.
The booking flow previously used `source_detail === 'dekel'` to determine whether to route
through the Voice Agent. With tenants, we route based on `tenant.name` and `tenant.monday_board_id`.

## Context

Plans 10-01 and 10-02 built:
- `tenants` table with `wa_phone_number_id`, `admin_phone`, `name`, `monday_board_id`, etc.
- `handleConversation(phone, messages, sock, tenant?)` with tenant-aware routing
- `buildSystemPrompt(..., tenant?)` using tenant config for personality/catalog
- Cloud webhook lookups tenant from `phoneNumberId` and threads it through

The `handleConversation` booking flow still used:
- `leadSource === 'dekel' || leadSource === 'dekel-voice-agent'` — hardcoded string check
- `config.mondayBoardIdDprisha` — hardcoded config key for Dekel board
- `'אלון מדקל לפרישה'` — hardcoded owner name for Dekel tenant

## Tasks

### Task 1: Replace hardcoded Dekel routing with tenant-based routing

**Type:** auto

In `handleConversation` booking flow (`[BOOK:...]` marker processing):
- Replace `isDekel = leadSource === 'dekel'...` with `useVoiceAgent = tenant?.name === 'דקל' && !!config.voiceAgentUrl`
  with a fallback to legacy source_detail check when no tenant
- Replace `isDekel ? 'אלון מדקל לפרישה' : getOwnerName()` with `tenant ? tenant.owner_name : getOwnerName()`
- Replace `config.mondayBoardIdDprisha` with `tenant?.monday_board_id ?? lead?.monday_board_id ?? config.mondayBoardIdDprisha`

**Files:** `src/ai/conversation.ts`

**Done when:** Booking flow uses tenant context, falls back gracefully to legacy mode.
