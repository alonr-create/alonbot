# Plan 1: Shell Command Blocklist & Input Validation

## Objective
Block dangerous shell commands via a centralized blocklist used by both the `shell` tool and cron script execution, validate all tool parameters with Zod schemas, restrict `auto_improve` file access, and sanitize web content from `browse_url`/`scrape_site` against prompt injection.

## Context
- The `shell` tool at `src/agent/tools.ts:278-284` passes `input.command` directly to `execSync()` with no filtering.
- The `fireCronJob` function at `src/cron/scheduler.ts:50` runs `parsed.script` via `execSync()` with no filtering.
- Tool parameters from Claude's JSON output have no runtime type validation — bad values crash deep in handlers.
- The `auto_improve` tool at `tools.ts:906-956` can modify any file including security-critical code.
- Web content from `browse_url` and `scrape_site` is returned to Claude without sanitization.
- Research provides complete blocklist patterns (`01-RESEARCH.md` section 1), Zod schemas (section 4), sanitization patterns (section 5), and auto_improve restrictions (section 4).
- User decisions: blocklist approach, not allowlist; auto_improve restricted to `system-prompt.ts` and `skills/`; sanitization is defense-in-depth.

## Task 1: Create shell blocklist utility

Create `src/utils/shell-blocklist.ts` with the `BLOCKED_SHELL_PATTERNS` array and `isShellCommandSafe()` function exactly as specified in `01-RESEARCH.md` section 1. The function must be pure (no side effects) and exported for future testing.

Key patterns to include:
- Filesystem destruction: `rm -rf /`, `mkfs`, `dd of=/dev/`
- Fork bombs: `:(){ ... };:`
- Remote code execution: `curl|bash`, `wget|sh`
- Privilege escalation: `chmod 777 /`, `chmod -R 777`
- System manipulation: `shutdown`, `reboot`, `iptables -F`
- Credential theft: `/etc/shadow`, `cat .ssh/id_`, `cat .env`
- Reverse shells: `nc -e`, `/dev/tcp/`, `mkfifo|nc`
- Eval injection: `eval $(`, `python -c ... socket`, `node -e ... child_process`
- History/log wiping: `history -c`, `> /var/log/`

Ensure the `cat .env` pattern is specific enough to avoid blocking `cat .env.example` — target `\bcat\b.*\.env\b` which matches `.env` at word boundary.

## Task 2: Create content sanitization utility

Create `src/utils/sanitize.ts` with `sanitizeWebContent()` as specified in `01-RESEARCH.md` section 5. Patterns to filter:
- "ignore previous instructions" variants
- Fake tool-use XML tags (`<tool_use>`, `<function_call>`)
- Zero-width character sequences (hidden text)
- Base64/eval code injection patterns
- Shell code blocks targeting destructive commands
- HTML comments (can contain hidden instructions)

Export the function for use in `tools.ts`.

## Task 3: Install Zod and add tool parameter schemas

Run `npm install zod`. Then add Zod schemas at the top of `src/agent/tools.ts` (after existing imports). Define schemas for high-risk tools as specified in `01-RESEARCH.md` section 4:

```typescript
import { z } from 'zod';
```

Schemas needed (at minimum):
- `shellSchema`: `{ command: z.string().min(1).max(10000) }`
- `writeFileSchema`: `{ path: z.string().min(1).max(500), content: z.string().max(500_000) }`
- `sendEmailSchema`: `{ to: z.string().email(), subject: z.string().min(1).max(500), body: z.string().min(1).max(50000) }`
- `deployAppSchema`: `{ project_dir: z.string().min(1), project_name: z.string().regex(/^[a-zA-Z0-9-]+$/).max(100).optional(), platform: z.enum(['vercel', 'railway']) }`
- `autoImproveSchema`: `{ action: z.enum(['list', 'read', 'edit']), file: z.string().max(500).optional(), search: z.string().max(10000).optional(), replace: z.string().max(50000).optional() }`
- `setReminderSchema`: `{ name: z.string().min(1).max(200), cron_expr: z.string().regex(/^[\d*,\/-\s]+$/).max(100), message: z.string().min(1).max(5000) }`
- `browseUrlSchema`: `{ url: z.string().url().max(2000) }`
- `mondayApiSchema`: `{ query: z.string().min(1).max(10000) }`
- `codeAgentSchema`: `{ task: z.string().min(1).max(10000), max_budget: z.number().min(0.1).max(10).optional(), model: z.string().max(50).optional(), working_dir: z.string().max(200).optional() }`
- `cronScriptSchema`: `{ name: z.string().min(1).max(200), cron_expr: z.string().regex(/^[\d*,\/-\s]+$/).max(100), script: z.string().min(1).max(10000), notify: z.boolean().optional() }`

Create a `TOOL_SCHEMAS` record mapping tool names to schemas.

## Task 6: Add validation wrapper in executeTool

At the top of `executeTool()` (line 263 in `src/agent/tools.ts`), before the cloud proxy check, add Zod validation:

```typescript
const schema = TOOL_SCHEMAS[name];
if (schema) {
  const result = schema.safeParse(input);
  if (!result.success) {
    const errors = result.error.issues.map(i => `${i.path.join('.')}: ${i.message}`).join('; ');
    return `Validation error: ${errors}`;
  }

}
```
